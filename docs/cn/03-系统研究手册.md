# domain-check 系统研究手册

本手册面向架构研究、二次开发和长期维护，重点回答三类问题：

- 系统如何组织能力与职责边界？
- 关键设计为什么成立，代价是什么？
- 如何做可重复、可解释、可行动的研究？

## 1. 新手先读：三句话理解系统

1. `domain-check` 是“域名可用性判断引擎”，不是域名注册系统。
2. 它优先走 RDAP（结构化协议），不行再回退 WHOIS（文本协议）。
3. 它的结果受外部网络与注册局状态影响，`UNKNOWN` 是正常工程状态之一。

## 2. 术语表（通俗解释）

| 术语 | 通俗解释 | 为什么重要 |
|---|---|---|
| RDAP | 新一代域名注册数据协议，返回结构化 JSON | 更稳定、可解析，适合自动化 |
| WHOIS | 传统查询协议，多为文本 | 覆盖广，但格式不统一 |
| FQDN | 完整域名，如 `myapp.com` | 判断对象的最小单位 |
| TLD | 顶级域后缀，如 `com`、`io` | 决定查询目标范围 |
| Bootstrap | 从 IANA 拉取注册表映射数据 | 提高后缀覆盖与准确性 |
| Fallback | 主流程失败后的兜底路径 | 提升整体可用性 |
| UNKNOWN | 当前证据不足无法判定 | 需要重试或人工复核 |
| Workspace | Rust 多 crate 工程结构 | 支撑“一核多端”复用 |
| MCP | 面向 AI 的工具调用协议 | 让 AI 可结构化调用能力 |

## 3. 研究对象定义

`domain-check` 采用“一核多端”结构：

- `domain-check-lib`：核心引擎和公共模型。
- `domain-check`：CLI 接入层。
- `domain-check-mcp`：MCP 接入层。

研究时建议把问题分三层：

- 协议层问题：RDAP / WHOIS 的数据质量与可达性；
- 工程层问题：并发、配置、输出、异常处理；
- 外部依赖问题：网络、注册局、服务波动。

## 4. 架构分层视图

| 层级 | 主要模块 | 核心职责 | 研究关注点 |
|---|---|---|---|
| 接入层 | CLI、MCP | 参数/请求接入和结果输出 | 可用性、接口一致性 |
| 编排层 | `DomainChecker` | 并发调度、回退策略、结果聚合 | 正确性、吞吐、可观测性 |
| 协议层 | `rdap.rs`、`whois.rs` | 与注册数据源交互 | 标准差异、容错、解析质量 |
| 注册表层 | `registry.rs` | TLD 映射、bootstrap、缓存 | 覆盖率、时效性、降级行为 |
| 配置层 | `config.rs` | 多源配置发现与合并 | 优先级、冲突处理、一致性 |
| 模型层 | `types.rs` | 统一数据结构 | 跨端兼容与字段稳定性 |

## 5. 关键设计与取舍

### 5.1 RDAP 优先，WHOIS 回退

为什么这样做：

- RDAP 是结构化数据，机器处理更稳定；
- WHOIS 覆盖更广，能弥补部分后缀的 RDAP 缺失。

代价：

- WHOIS 文本标准化差，解析策略更复杂；
- 外部服务波动会引入不确定结果。

研究建议：

- 将“协议失败”和“网络失败”分开统计；
- 分 TLD 观察 `UNKNOWN` 占比，识别结构性薄弱区。

### 5.2 一核多端复用

为什么这样做：

- CLI、Library、MCP 共用引擎，减少重复实现；
- 同一数据模型降低跨端语义偏差。

代价：

- 核心逻辑改动会同时影响多个入口；
- 变更需要更强的回归测试和文档同步。

研究建议：

- 建立跨端回归用例；
- 文档中显式标注“当前实现行为”。

### 5.3 配置多源合并

为什么这样做：

- 兼容本地项目、用户目录和自动化环境；
- 便于团队将默认参数标准化。

代价：

- 排障时不容易判断最终生效值来源；
- 文档容易和实现产生漂移。

研究建议：

- 输出配置优先级矩阵；
- 对关键字段做显式对照实验。

## 6. 研究方法论（可直接复用）

### 6.1 三段式实验法

1. 定义基线：固定输入、参数、网络条件。
2. 单变量实验：每次只改一个变量。
3. 分层归因：协议、工程、外部依赖分别解释。

### 6.2 证据收集清单

- 命令快照：完整命令和参数；
- 结果快照：原始 JSON / CSV；
- 环境快照：系统、版本、网络条件；
- 解释快照：差异原因和判定依据。

### 6.3 结论质量门槛

高质量结论应满足：

- 可重复：他人按步骤可复现主要结论；
- 可解释：结论与证据链一致；
- 可行动：能转化为配置、流程或代码改进。

## 7. 典型研究课题

### 课题 A：并发与稳定性平衡

目标：找到稳定吞吐区间，不追求单次峰值。

变量：

- `--concurrency`（10 / 20 / 40 / 80）
- 批量规模（100 / 1000）
- 输出模式（默认 / `--batch`）

指标：

- 总耗时；
- `UNKNOWN` 比率；
- 平均单域耗时。

### 课题 B：bootstrap 依赖策略

目标：评估 `--no-bootstrap` 对覆盖率和稳定性的影响。

变量：

- 默认模式；
- 禁用 bootstrap；
- 不同 TLD 组合（gTLD + ccTLD）。

指标：

- 有效判定率；
- `UNKNOWN` 波动；
- 结果可复现性。

### 课题 C：MCP 工具链可用性

目标：验证 AI 客户端场景中的结构化调用稳定性。

变量：

- 单域与批量请求；
- 不同客户端配置；
- 边界输入和异常输入。

指标：

- 调用成功率；
- 返回结构一致性；
- 错误信息可操作性。

## 8. 新手最容易混淆的 5 个点

1. `UNKNOWN` 不是 bug 结论，而是证据状态。
2. 查询速度不只由本地并发决定，也受外部服务限制。
3. “可解析 DNS”不等于“域名可注册”。
4. 配置生效顺序会影响实验结果解释。
5. 文档示例可运行不代表在所有网络环境结果一致。

## 9. 开发与发布研究切面

CI / Release 流程对研究的意义：

- `fmt + clippy + test + audit` 提供质量底线；
- 多平台矩阵验证跨平台一致性；
- 发布流程保证版本与产物可追踪。

研究动作建议：

- 关键改动先过本地基线再看 CI；
- 发布后执行一次回归抽样；
- 对安全审计结果做周期复盘。

## 10. 文档与实现一致性检查

建议每次版本迭代执行一次检查：

- 环境变量列表与代码解析范围是否一致；
- 输出相关配置是否在运行路径真实生效；
- 配置搜索路径是否与文档一致。

输出建议：

- 标注“当前行为”与“规划行为”；
- 对存在差异项给出临时使用建议。

## 11. 演进方向建议

- 建立文档-实现自动对齐检查；
- 增强运行时可观测性；
- 强化 CLI / Library / MCP 统一回归测试；
- 将核心研究课题沉淀为长期基准测试。

## 12. 研究报告模板

每份研究报告建议包含：

1. 问题定义与背景；
2. 实验设计（变量、对照、指标）；
3. 执行记录（命令、配置、环境）；
4. 结果与归因；
5. 风险与限制；
6. 可执行改进项。

按此模板输出，可显著提升跨团队沟通与决策效率。
