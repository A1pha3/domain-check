# domain-check 系统学习路径

本学习路径面向团队培养和个人进阶，按“会使用 → 会自动化 → 会研究与改进”三层设计，强调“每个阶段都能产出可验收结果”。

## 1. 学习目标

完成本路径后，你应具备以下能力：

- 能独立完成域名批量检查、候选生成与结果导出。
- 能解释 RDAP 与 WHOIS 的分工、边界和不确定性来源。
- 能在 CI / AI 工具链中稳定集成 `domain-check`。
- 能阅读核心模块并提出有证据的改进建议。

## 2. 学前准备

建议具备以下基础：

- 命令行基础：会执行命令、重定向、管道。
- 数据基础：理解 JSON / CSV 的结构和用途。
- 工程基础：理解“可复现命令”和“参数固定化”的价值。

如果基础薄弱，建议先完成两项热身：

1. 运行 `domain-check --help` 并读完参数分组。
2. 用 `--json` 输出一批结果并用 `jq` 做一次筛选。

## 3. 学习分层

| 层级 | 角色定位 | 核心能力 | 阶段产出 |
|---|---|---|---|
| L1 基础使用者 | 日常执行任务 | 命令、输出、常见问题处理 | 一份完整检查结果与说明 |
| L2 进阶使用者 | 自动化建设者 | 配置治理、CI 集成、结果策略 | 一条可复用流水线脚本 |
| L3 维护研究者 | 系统参与者 | 架构理解、实验设计、演进建议 | 一份研究报告或改进提案 |

## 4. 分阶段学习计划

### 阶段 A：建立最小闭环（L1）

#### 必学主题

- 输入方式：参数、文件、模式、前后缀。
- 输出方式：默认、pretty、JSON、CSV、info。
- 关键约束：互斥参数、并发范围、批量策略。

#### 实践任务

1. 用 `--preset startup --pretty` 完成一次品牌检查。
2. 用 `--pattern` + `--dry-run` 生成候选并做人工筛选。
3. 用 `--batch --json --yes` 输出机器可读结果并保存到文件。

#### 验收标准

- 能清楚解释每条命令的输入、输出和目的。
- 能说明 `UNKNOWN` 不等于可用。
- 能复现同一任务并得到结构一致结果。

#### 自测题

- 为什么 `--streaming` 不能和 `--json/--csv` 同时使用？
- 为什么自动化任务建议加 `--yes`？
- 什么情况下应降低并发而不是继续重试？

### 阶段 B：形成自动化能力（L2）

#### 必学主题

- 配置优先级与环境变量注入策略。
- CI 任务中的参数固定化与输出稳定化。
- 结果治理：重试、阈值、告警、审计留档。

#### 实践任务

1. 写一条可在 CI 运行的 JSON 输出命令。
2. 设计 `UNKNOWN` 的分层重试规则并给出阈值。
3. 用配置文件沉淀团队默认行为（并发、预设、超时）。

#### 验收标准

- 本地和 CI 结果结构一致。
- 流程中有明确失败处理分支。
- 可解释每个关键参数的设置理由。

#### 自测题

- 何时优先用命令行参数，何时用环境变量？
- 如何保证脚本在非 TTY 环境不被交互阻塞？
- 哪些字段必须保留用于后续审计？

### 阶段 C：进入架构与研究层（L3）

#### 必学主题

- 核心模块职责：`DomainChecker`、协议层、注册表层、配置层。
- RDAP-first + WHOIS fallback 的设计意义。
- 质量门禁与发布链路（fmt / clippy / test / audit / release）。

#### 实践任务

1. 画出一次 CLI 请求的数据流。
2. 对比启用 / 禁用 bootstrap 的结果差异并归因。
3. 输出一份“可演进点清单”，至少含可靠性或可维护性视角。

#### 验收标准

- 能区分“外部不确定性”与“实现缺陷”。
- 能将结论关联到可执行改进动作。
- 能向新人讲清楚核心设计取舍。

#### 自测题

- 为什么同一引擎可同时支持 CLI、Library、MCP？
- 哪些改动应优先放在核心库而不是入口层？
- 你会先补测试、补文档，还是改逻辑？为什么？

## 6. 训练营模板（团队可复用）

### 6.1 单人模板（1 人）

- 任务 1：完成 L1 三个实践任务并记录命令和输出。
- 任务 2：把一个任务脚本化，加入错误处理。
- 任务 3：写一页“系统边界与我的理解”。

### 6.2 小组模板（3-5 人）

- 角色分工：使用组、自动化组、研究组。
- 统一输入：同一批候选域名、同一验收标准。
- 统一输出：命令模板库、故障手册、改进提案清单。

## 7. 常见误区与纠偏

- 误区：把一次 `UNKNOWN` 当作最终结论。  
  纠偏：纳入重试和人工复核，不直接做业务决策。

- 误区：自动化里依赖交互式默认行为。  
  纠偏：显式使用 `--batch --json --yes`。

- 误区：把所有波动都归因于代码实现。  
  纠偏：先分离网络、注册局和协议标准化差异因素。

## 8. 学习成效评估

建议按四个维度评分（每项 25 分）：

- 任务完成度：是否达成阶段任务并可复现。
- 结果可信度：是否保留证据、重试策略与解释。
- 表达清晰度：是否能清楚向他人讲解原理。
- 迁移能力：是否能将方法应用到新场景。

达到 80 分以上，可视为“可独立承担该层级任务”。
