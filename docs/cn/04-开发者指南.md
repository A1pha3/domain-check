# domain-check 开发者指南

本指南面向参与 `domain-check` 开发、维护与发布的工程人员，覆盖从本地开发到 CI / Release 的完整流程。

## 1. 开发目标

完成本指南后，你应能：

- 在本地独立完成构建、测试、静态检查与文档检查。
- 理解 workspace 与三大 crate 的职责边界。
- 以低风险方式提交代码并通过 CI。
- 参与版本发布与问题定位。

## 2. Rust 环境安装与验证（新手必读）

### 2.1 macOS 推荐安装方式

使用官方 `rustup` 安装：

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

安装后重启终端，或执行：

```bash
source "$HOME/.cargo/env"
```

### 2.2 检查是否安装成功

```bash
rustc --version
cargo --version
rustup --version
```

建议确认已安装 stable 工具链：

```bash
rustup show
```

### 2.3 常用环境准备

安装代码格式组件：

```bash
rustup component add rustfmt
```

安装静态检查组件：

```bash
rustup component add clippy
```

如果你在 Apple Silicon（M 系列）机器开发，默认目标已可满足本仓库本地开发需求。

## 3. 开发环境建立（从 0 到可提交）

### 3.1 克隆与首次检查

```bash
git clone https://github.com/saidutt46/domain-check.git
cd domain-check
cargo check --workspace
```

### 3.2 建议工具链

- 编辑器：VS Code 或 JetBrains Rust 插件。
- Git 工具：`git` 必需，`gh` 可选。
- JSON 处理：建议安装 `jq`，便于调试自动化输出。

### 3.3 目录快速认知

- `domain-check-lib`：核心业务逻辑。
- `domain-check`：CLI 参数和输出。
- `domain-check-mcp`：MCP 工具层。
- `docs/`：文档与说明。

建议先跑一次 CLI 命令感知行为：

```bash
cargo run -p domain-check -- example.com
```

## 4. 仓库结构与职责

项目采用 Rust workspace，核心目录如下：

- `domain-check-lib`：核心引擎与公共类型。
- `domain-check`：CLI 入口与交互输出。
- `domain-check-mcp`：MCP 工具服务。
- `docs/`：用户与运维文档。
- `.github/workflows/`：CI、发布与安全流程。

理解这层结构的意义在于：行为逻辑优先落在 `domain-check-lib`，入口层只负责接入与展示。

## 5. 日常开发命令

建议将以下命令作为本地提交前基线：

```bash
# 构建
cargo build --workspace

# 测试
cargo test --workspace

# 静态检查
cargo clippy --workspace --all-targets --all-features -- -D warnings -A clippy::uninlined_format_args

# 格式检查
cargo fmt --all --check
```

当你修改跨 crate 公共逻辑时，优先执行 `--workspace` 级别命令，避免局部通过而全局失败。

## 6. 开发流程（怎么开发）

### 6.1 标准开发闭环

1. 建分支：`git checkout -b feat/your-topic`
2. 改代码：优先在对应职责层修改。
3. 先自测：`cargo test --workspace`
4. 再质检：`cargo clippy ...` 与 `cargo fmt ...`
5. 补文档：行为变化必须同步文档。
6. 提交 PR：附上变更动机、影响范围、验证证据。

### 6.2 一个最小开发示例

假设你要新增一个 CLI 参数校验：

1. 在 `domain-check/src/main.rs` 增加参数语义或校验逻辑。
2. 在 `domain-check-lib` 增加必要配置或类型支持。
3. 添加对应测试或更新已有测试。
4. 更新 `docs/CLI.md` 与 `docs/cn` 文档中的对应段落。
5. 执行本地基线命令后提交。

### 6.3 改动落点建议

- 业务规则变化：优先改 `domain-check-lib`。
- CLI 展示或参数交互：改 `domain-check`。
- AI 工具协议适配：改 `domain-check-mcp`。

## 7. 代码变更路径

建议流程：

1. 从 `main` 拉新分支，避免直接在主分支开发。
2. 将“行为变更”和“重构整理”分开提交。
3. 用户可感知行为变化时同步更新文档。
4. 行为变化必须补充或更新测试。

分支命名可采用：

- `fix/*`
- `feat/*`
- `docs/*`
- `chore/*`

## 8. 质量门禁与 CI 对齐

CI 关注的核心维度：

- 格式：`cargo fmt --check`
- 静态检查：`cargo clippy -D warnings`
- 测试：workspace、all-features、doc tests、集成测试
- 安全：`cargo audit` 与依赖审查
- 兼容性：多操作系统 + 多 Rust 版本矩阵

开发建议：

- 本地先跑一遍核心基线，再推送分支。
- 对有风险变更增加最小复现用例。
- 对外部依赖波动引发的问题，记录失败证据再修复。

## 9. 核心模块开发要点

### 7.1 `domain-check-lib`

重点模块：

- `checker.rs`：检查编排、并发控制、回退策略。
- `protocols/rdap.rs`：RDAP 请求与解析。
- `protocols/whois.rs`：WHOIS 请求与解析。
- `protocols/registry.rs`：TLD 映射、bootstrap、缓存。
- `config.rs`：配置发现、合并与环境变量解析。

开发原则：

- 优先保持 `DomainResult` 等公共模型稳定。
- 涉及协议行为改动时，补充边界场景测试。
- 明确区分“协议不确定性”和“程序错误”。

### 7.2 `domain-check`（CLI）

开发关注点：

- 参数语义与互斥规则的一致性。
- 输出模式的稳定性（默认 / pretty / JSON / CSV）。
- 自动化场景的非交互行为（`--yes`、非 TTY）。

### 7.3 `domain-check-mcp`

开发关注点：

- 工具输入输出结构稳定。
- 错误信息可供 AI 客户端消费。
- 与核心库行为保持一致，不重复实现业务逻辑。

## 10. 调试与排障

建议使用“先缩小再定位”的方式：

1. 先用最小输入复现问题。
2. 固定并发与输出模式，排除噪声。
3. 对比 `--no-bootstrap` 与默认行为差异。
4. 判断问题来源：网络、注册局、协议解析或业务逻辑。

若涉及 MCP 调试，可优先验证工具输入输出，再检查客户端配置。

## 11. 发布与版本协作

发布链路关键点：

- 由版本 tag 触发发布流程。
- 校验 tag 与 `Cargo.toml` 版本一致。
- 构建多平台二进制并生成发布产物。
- 依次发布 crate，并执行发布后验证。

协作建议：

- 发布前完成文档与变更日志同步。
- 对破坏性变化给出迁移说明。
- 将“发布后验证结果”纳入发布记录。

## 12. 文档维护规范

建议遵循：

- 根 `README.md` 保持入口简洁。
- 深度细节放入 `docs/` 下专题文档。
- 中文文档放入 `docs/cn` 并保持导航可达。
- 文档中的版本、参数、行为以当前实现为准。

## 13. 新手常见问题

### 13.1 为什么我本地能跑，CI 却失败

常见原因：

- 本地未执行 `fmt` 或 `clippy`；
- 只测了单 crate，未测 `--workspace`；
- 修改了行为但没有同步测试或文档。

### 13.2 什么时候必须补测试

只要是行为变化、边界处理变化、输出结构变化，都建议补测试。

### 13.3 什么时候必须补文档

只要用户可感知行为变化，就必须更新相关文档。

## 14. 提交前检查清单

- 已通过 `build + test + clippy + fmt`。
- 行为变更已补测试与文档。
- 输出与错误信息不破坏现有消费方。
- 关键改动已说明风险与回滚策略。

满足以上清单后再提交 PR，可显著降低返工成本。
