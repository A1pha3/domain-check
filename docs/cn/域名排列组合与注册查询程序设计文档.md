# Python 域名排列组合与注册查询程序设计文档

> 文档类型：系统设计文档
>
> 目标读者：具备 Python 基础、希望批量筛选可用域名的开发者
>
> 文档级别：Level 3⭐⭐⭐

---

## 1. 背景与目标

你希望实现一个 Python 程序：输入一组给定单词，自动生成可控规模的域名候选，再批量查询这些候选域名是否已注册，并输出可用于后续筛选的结果。

核心目标如下：

- 基于给定单词进行排列组合，生成高质量候选
- 支持多个顶级域（TLD）批量查询
- 区分“已注册 / 未注册 / 未知（查询失败）”
- 在保证查询效率的同时，控制速率与合规风险
- 结果可导出为 CSV / JSON，支持后续人工与自动筛选

---

## 2. 范围定义

### 2.1 功能范围（In Scope）

- 输入单词列表（命令行参数、文件、配置文件）
- 候选生成（排列、组合、连接符策略、长度约束）
- 域名规范化（小写、字符清洗、去重）
- 多 TLD 扩展（如 `.com`、`.net`、`.io`）
- 域名状态查询（优先 RDAP，回退 WHOIS）
- 结果持久化（CSV、JSON）
- 基础统计（总数、可用数、失败数、耗时）

### 2.2 非功能范围（Out of Scope）

- 域名自动注册购买
- 规避查询限流或反爬机制
- 大规模分布式爬取平台
- 商业级风控绕过逻辑

---

## 3. 关键需求

### 3.1 功能性需求

1. 支持输入 N 个单词，按配置生成候选域名标签（label）。
2. 支持组合策略：
   - 全排列（permutation）
   - 固定长度组合（combination）
   - 可选连接符（空字符串、`-`）
3. 支持候选过滤：
   - 标签长度限制（1-63）
   - 字符合法性（`a-z`、`0-9`、`-`）
   - 不允许首尾连字符
4. 支持按并发数与速率限制发起查询。
5. 对每个完整域名输出状态、证据和时间戳。

### 3.2 非功能性需求

- **正确性**：同一查询批次内不重复请求相同域名。
- **性能**：1 万级候选可在可接受时间内完成（依赖网络与注册局响应）。
- **稳定性**：网络抖动、超时、临时失败可重试。
- **可观测性**：运行时输出进度与阶段统计。
- **可扩展性**：可新增查询后端（如特定注册商 API）。

---

## 4. 总体架构

采用“流水线 + 插件化后端”架构：

1. `InputLoader`：读取输入与配置
2. `CandidateGenerator`：生成候选标签
3. `CandidateFilter`：过滤与去重
4. `DomainExpander`：拼接 TLD 得到 FQDN
5. `AvailabilityChecker`：查询状态（RDAP/WHOIS）
6. `ResultWriter`：写入 CSV/JSON
7. `Reporter`：输出统计与摘要

数据流：

`输入单词 -> 候选标签 -> 过滤去重 -> 域名全名 -> 查询状态 -> 结果落盘 -> 汇总报告`

## 5. 模块设计

### 5.1 输入与配置模块

**输入来源**

- 命令行：`--words alpha,beta,gamma`
- 文件：`--words-file words.txt`（一行一个词）
- 配置文件：`config.toml`

**关键配置项**

- `min_terms` / `max_terms`：单个候选包含的单词数量范围
- `joiners`：连接符列表（如 `["", "-"]`）
- `tlds`：TLD 列表（如 `["com", "net", "io"]`）
- `max_candidates`：候选上限，防止组合爆炸
- `concurrency`：并发数
- `qps`：每秒请求数上限
- `timeout_sec`：单请求超时
- `retry_times`：重试次数

### 5.2 候选生成模块

支持两类生成策略：

1. **排列（Permutation）**  
   例如词集 `[alpha, hub, lab]`，取 2 个词时生成：
   - `alphahub`、`hubalpha`
   - `alphalab`、`labalpha`
   - `hublab`、`labhub`

2. **组合（Combination）**  
   仅保留顺序无关集合，例如：
   - `alpha+hub`、`alpha+lab`、`hub+lab`
   再按 joiner 规则构造字符串。

建议默认流程：

- 先组合（控制规模）
- 再可选启用排列（用于补充高优先级短词）

### 5.3 候选过滤模块

过滤规则：

- 全转小写
- 非法字符剔除或替换
- 连续连字符压缩为单个
- 首尾连字符剔除
- 长度在 1-63
- 使用集合去重

同时计算一个“优先级分”用于结果排序：

- 长度越短分越高
- 无连字符分更高
- 命中用户关键词分更高

### 5.4 查询模块（AvailabilityChecker）

采用分层查询策略：

1. **首选 RDAP**  
   - 结构化响应、可解析性高
   - 如果返回“对象不存在”可判定“可能未注册”
2. **回退 WHOIS**
   - 对不支持 RDAP 的域后缀进行补充
   - 解析不同注册局返回文本中的关键字段
3. **结果状态模型**
   - `registered`
   - `available`
   - `unknown`（超时、限流、解析失败）

证据字段建议保留：

- `source`：`rdap` / `whois`
- `raw_hint`：关键响应片段
- `checked_at`：查询时间

### 5.5 并发与限流模块

建议基于 `asyncio`：

- `Semaphore` 控制最大并发
- 令牌桶或漏桶控制 QPS
- 指数退避重试（如 `0.5s -> 1s -> 2s`）

避免策略：

- 无限制并发打满目标服务
- 单域名短时间重复重试

### 5.6 结果输出模块

输出格式一：CSV（便于表格筛选）

字段建议：

- `domain`
- `status`
- `source`
- `checked_at`
- `score`
- `error`

输出格式二：JSON（便于程序继续处理）

包含：

- 配置快照
- 统计信息
- 明细结果数组

### 6.1 候选标签模型

- `label: str`
- `tokens: list[str]`
- `joiner: str`
- `score: float`

### 6.2 域名查询结果模型

- `domain: str`
- `status: Literal["registered", "available", "unknown"]`
- `source: str`
- `checked_at: datetime`
- `raw_hint: str | None`
- `error: str | None`
- `duration_ms: int`

## 7. 执行流程设计

### 7.1 主流程

1. 解析参数与配置
2. 加载并清洗单词
3. 生成候选并去重
4. 拼接 TLD 形成待查域名
5. 并发查询域名状态
6. 写入结果文件
7. 输出汇总统计

### 7.2 异常流程

- 配置非法：立即终止并输出可修复建议
- 网络超时：按重试策略处理，超限后标记 `unknown`
- 结果写盘失败：保留内存结果并提示重试导出

## 8. 命令行接口设计（CLI）

示例：

```bash
uv run python -m domain_probe \
  --words-file words.txt \
  --tlds com,net,io \
  --min-terms 2 \
  --max-terms 3 \
  --joiners "",- \
  --mode combination \
  --concurrency 20 \
  --qps 10 \
  --timeout-sec 5 \
  --retry-times 2 \
  --output results/domains.csv
```

建议参数分组：

- 输入参数：`--words`、`--words-file`
- 生成参数：`--mode`、`--min-terms`、`--max-terms`、`--joiners`
- 查询参数：`--tlds`、`--concurrency`、`--qps`、`--timeout-sec`、`--retry-times`
- 输出参数：`--output`、`--output-json`

## 9. 风险与合规

### 9.1 技术风险

- **组合爆炸**：词数增大导致候选数量指数上升  
  对策：强制 `max_candidates` + 优先级截断。

- **查询结果不一致**：不同注册局响应标准不同  
  对策：状态中保留 `unknown` 与证据字段，不做过度推断。

- **网络抖动**：超时与临时失败增多  
  对策：合理重试 + 失败统计分层展示。

### 9.2 合规建议

- 遵守公开查询接口的使用条款与速率限制
- 不对单一服务进行高频冲击
- 将程序定位为“可用性初筛工具”，最终注册前再人工复核

## 16. 术语补充（新手友好版）

### 16.1 什么是“组合爆炸”

词数量和组合长度上升后，候选数量会快速增长。  
例如从 5 个词选 2 个，与从 50 个词选 2 个，规模不是线性增加，而是急剧上升。

### 16.2 什么是“限流”

限流是主动控制请求发送速度，避免把目标服务压垮，也降低被限频的风险。

### 16.3 为什么要保留 `unknown`

真实网络环境中存在超时、暂时失败、解析异常。  
保留 `unknown` 可以防止“误判为可用”或“误判为已注册”。

### 16.4 为什么“可解析”不等于“可注册”

DNS 可解析只说明某个名称可被解析，不直接等价于域名注册状态。  
注册状态需看注册数据源（RDAP / WHOIS）给出的证据。

## 17. 从设计到实现的建议路径

如果你要把本文方案落地成可运行系统，建议按以下顺序推进：

1. 先做最小可用版本：生成 + 查询 + CSV 输出；
2. 再补工程能力：并发、限流、重试、JSON 输出；
3. 再做可维护能力：配置体系、测试体系、日志与指标；
4. 最后做规模优化：缓存、任务拆分、可观测性增强。

这一路径可帮助你在“快速可用”和“长期可维护”之间取得平衡。
