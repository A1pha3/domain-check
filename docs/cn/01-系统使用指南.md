# domain-check 系统使用指南

本指南覆盖 `domain-check` 的 CLI、配置、自动化和 MCP 接入实践，面向“第一次接触也能上手”的目标编写。

## 1. 你将学会什么

读完本指南，你可以：

- 独立完成单域名与批量域名检查。
- 使用预设、模式、前后缀生成候选域名。
- 用 JSON / CSV 把结果接入自动化流程。
- 理解常见错误和结果波动的原因。

## 2. 系统定位（先建立整体认识）

`domain-check` 是统一域名探测系统，提供三种使用形态：

- CLI：给人使用的命令行入口。
- Library：给 Rust 代码调用的库接口。
- MCP Server：给 AI 客户端调用的工具服务。

核心检查逻辑一致：优先使用 RDAP（结构化数据），失败时回退 WHOIS（文本数据）。

## 3. 五分钟快速上手

### 3.1 安装

macOS 推荐：

```bash
brew install domain-check
```

通用方式：

```bash
cargo install domain-check
```

### 3.2 验证安装

```bash
domain-check --help
```

如果命令可执行，说明安装成功。

### 3.3 最小可运行命令

```bash
domain-check example.com
```

### 3.4 三个最常用命令模板

```bash
# 1) 品牌初筛
domain-check mybrand --preset startup --pretty

# 2) 先生成候选再人工确认（不发起查询）
domain-check --pattern "app\d" -t com --dry-run

# 3) 自动化批量输出
domain-check --file domains.txt --batch --json --yes
```

## 4. 核心概念（新手必读）

### 4.1 什么是 FQDN

FQDN（Fully Qualified Domain Name）是完整域名，例如 `myapp.com`。  
在命令中你既可以输入完整域名，也可以输入基础名（如 `myapp`）再配合 TLD 扩展。

### 4.2 什么是 TLD

TLD（Top-Level Domain）是顶级域后缀，如 `com`、`io`、`org`。  
系统支持三种 TLD 来源：

- `--tld`：手动指定；
- `--preset`：按预设指定；
- `--all`：遍历已知后缀。

### 4.3 为什么会出现 `UNKNOWN`

`UNKNOWN` 不是“可用”，也不是“已注册”，而是“当前证据不足以判断”。  
常见原因：

- 网络超时；
- 注册局响应不稳定；
- 协议返回内容不标准。

正确做法是重试并人工复核。

## 5. 输入模型与参数组合

### 5.1 输入来源

- 直接参数：`domain-check myapp`
- 文件输入：`--file domains.txt`
- 模式生成：`--pattern "go?\d"`
- 前后缀组合：`--prefix get,my --suffix hub,ly`

### 5.2 TLD 选择建议

三选一即可：

- `--tld com,io,dev`
- `--preset startup`
- `--all`

尽量避免一次命令中混用多种 TLD 来源，降低歧义。

### 5.3 互斥关系（高频踩坑点）

- `--json` 与 `--csv` 互斥。
- `--streaming` 不能和 `--json/--csv` 同用。
- `--batch` 与 `--streaming` 不要同时启用。

## 6. 输出模型与消费方式

### 6.1 默认文本

适合人工快速查看。

### 6.2 pretty 输出

`--pretty` 会分组展示，适合评审会议和讨论场景。

### 6.3 JSON 输出

`--json` 适合脚本和平台集成，可直接 `pipe` 给 `jq`。

### 6.4 CSV 输出

`--csv` 适合导入表格、BI 或运营分析工具。

### 6.5 详细信息

`--info` 可附加注册商、日期、状态码等信息，适合尽调场景。

## 7. 常见工作流（可直接复制）

### 7.1 品牌可用性初筛

```bash
domain-check mybrand --preset startup --pretty
```

### 7.2 全后缀审计

```bash
domain-check mybrand --all --batch --json --yes > audit.json
```

### 7.3 CI 批量筛选可用域名

```bash
domain-check --file required-domains.txt --batch --json --yes | jq '.[] | select(.available==true)'
```

### 7.4 网络受限场景

```bash
domain-check mybrand -t com,org,io --no-bootstrap --batch
```

## 8. 配置系统（从“能用”到“好用”）

### 8.1 配置文件查找顺序

1. `./domain-check.toml` 或 `./.domain-check.toml`
2. `~/.domain-check.toml` 或 `~/domain-check.toml`
3. `~/.config/domain-check/config.toml`

### 8.2 推荐配置模板

```toml
[defaults]
concurrency = 25
preset = "startup"
pretty = true
timeout = "8s"
bootstrap = true

[custom_presets]
my_stack = ["com", "io", "dev", "app"]

[generation]
prefixes = ["get", "my"]
suffixes = ["hub", "ly"]
```

### 8.3 环境变量（当前实现已支持）

- `DC_CONCURRENCY`
- `DC_PRESET`
- `DC_TLD`
- `DC_PRETTY`
- `DC_TIMEOUT`
- `DC_WHOIS_FALLBACK`
- `DC_BOOTSTRAP`
- `DC_DETAILED_INFO`
- `DC_JSON`
- `DC_CSV`
- `DC_FILE`
- `DC_CONFIG`
- `DC_PREFIX`
- `DC_SUFFIX`

实践建议：命令行参数用于“本次任务明确意图”，环境变量用于“团队默认值注入”。

## 9. 自动化与脚本实践

### 9.1 标准非交互命令

```bash
domain-check --file domains.txt --batch --json --yes
```

### 9.2 重试策略建议

- 仅对 `UNKNOWN` 做重试；
- 第一轮重试保持参数不变；
- 第二轮重试可降低并发；
- 保留原始结果文件用于审计。

### 9.3 stdout / stderr 分离

- stdout：结果数据通道；
- stderr：日志与提示通道；
- CI 解析时仅消费 stdout。

## 11. 故障排查清单

### 11.1 没有结果或结果异常

- 检查参数是否互斥；
- 检查输入文件格式与空行；
- 降低并发到 `10` 验证稳定性。

### 11.2 大批量任务耗时明显增加

- 检查网络与注册局响应；
- 调整 `--concurrency`；
- 使用 `--batch --json` 减少终端渲染开销。

### 11.3 同一域名多次结果不一致

- 区分外部服务波动和本地参数变化；
- 固定参数后重复执行；
- 对关键域名进行人工复核。

## 12. 生产落地建议

- 把高频命令固化为团队脚本模板。
- 关键任务显式写出所有关键参数。
- 将 `UNKNOWN` 纳入待复核清单，不直接进入决策结果。
- 每次版本升级后复查配置和流程文档。
